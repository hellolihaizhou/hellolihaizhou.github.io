复现步骤：测试机开启双WiFi且连接测试路由主5G，辅2.4G
问题现象：主wifi漫游后不久，辅WiFi出现断连
问题概率：必现

辅wifi为什么会断连？

先看下辅wifi为何会断连，从测试同事提供的视频来看时间点大概是15:50:07左右

step1:

看下这个时间点的辅wifi状态机
tip: 过滤的关键字"WifiClientModeImpl:",打印不同时间点的wifi状态转移变化

step2:

看下当前的信号强度以及断线reason值
//搜索关键字”NETWORK_DISCONNECTION_EVENT”, 代表网络断开的标志性日志，信号值-21说明信号强度很好，reason为0

这个reason 0其实没有多大的参考意义，对应的802.11 Deauth Reason Codes表格中给出的解释是正常操作，具体的reason需要看fw打印的reason

step3:

确认下辅wifi是底层断线还是上层断线
直接看nl80211收到断线的消息早于wpa_supplicant: wlan1可以说明是底层掉线

//logcat

step4：

到这里我们只知道是底层断线的，具体原因还没找到，此时需要看下driver log以及fw log

//driver log

包括roam scan也没有，是因为辅wifi底层不支持漫游，此时信号-25很好

beacon miss的发生不一定会立刻掉线，达到一定数量才会触发掉线吧?
接下来在fw log中证实下这个猜想

这里的final_bmiss_detected ，first_bmiss_detected 解释对应如下

gRoamBmissFirstBcnt=10
After consecutive number of beacons missed as configured by gRoamBmissFirstBcnt, the system stays alive (no power collapse) t
o receive beacons until final BMISS (consecutive number of beacons missed as configured by gRoamBmissFinalBcnt) 

gRoamBmissFinalBcnt=20
After consecutive number of beacons missed as configured by gRoamBmissFirstBcnt, 
the system stays alive (no power collapse) to receive beacons until final BMISS (consecutive number of beacons missed as configured by gRoamBmissFinalBcnt)

If “p”(gRoamBmissFirstBcnt) beacons are missed in a row, roaming module will perform roaming scan and create a channel map.
If “q”(gRoamBmissFinalBcnt) more beacons are missed, it results in “final beacon miss”.

That is a trigger for roaming. If we don’t find the candidate, this will lead to disconnection.

大概意思是有连续的10个beacon missed发生了，这个时候不会断线，会触发漫游并会继续接收beacon。

如果在前面连续十个beacon missed之后又发生连续20个beacon missed的话，则会进入final beacon miss ，这个时候如果还没有找到候选AP的话，会最终导致断线
所以到这里明白了fw确实总计发生了30次beacon miss，并且由于是辅wifi不支持漫游的缘故，最终以断线收场

step5：

可是beacon miss的原因是什么呢？前面已经分析过断开时信号是-25说明不是信号弱导致，同时也看到了在超时掉线前sta有发probe request即空帧探测AP，没有收到AP的ACK，看上去大概率问题出在AP
但是测试同事提的这个问题是在主wifi漫游后才会出现辅wifi掉线，这和主wifi漫游有关系？和channel有关系？
